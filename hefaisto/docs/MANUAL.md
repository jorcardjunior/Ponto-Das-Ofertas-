## A Origem do Hefaisto

O Hefaisto nasceu de uma frustração real: **agências de marketing perdem tempo demais entre a ideia e a entrega**.

O cliente aprova o design de uma landing page pela manhã e pergunta "já está no ar?" à tarde. Entre essas duas perguntas, há um abismo de tarefas técnicas — programação, testes, configuração de integrações, deploy — que consome horas, dias, às vezes semanas.

O Hefaisto começou como **Hefaisto** (anteriormente Synkra Hefaisto), um framework com 13 agentes AI, 14 workflows e 6 portões de qualidade. Funcionava, mas era **burocrático demais**. Pedir uma correção simples exigia criar documentos, passar por múltiplas validações e coordenar agentes que nem sempre eram necessários.

A transformação para **Hefaisto** foi guiada por um princípio: **autonomia sem burocracia**.

**O que mudou:**

| Antes (Hefaisto) | Depois (Hefaisto) |
|:---|:---|
| 13 agentes genéricos | 5 especialistas com expertise real de mercado |
| 14 workflows diferentes | 1 ciclo autônomo que se adapta |
| 6 portões de qualidade obrigatórios | 1 check automático integrado |
| Precisava digitar comandos | Basta descrever o que precisa |
| Agentes sem conhecimento de marketing | Cada agente carrega metodologias dos melhores profissionais do mercado |

O resultado é um framework que **pensa como uma agência de marketing** e **executa como uma equipe de desenvolvimento sênior**.

## O que é o Hefaisto?

### A explicação simples

Imagine que você pudesse contratar 5 profissionais especializados que trabalham 24 horas por dia, nunca pedem férias, e entendem tanto de marketing quanto de tecnologia. É exatamente isso que o Hefaisto faz.

**O Hefaisto é um framework que coordena 5 agentes de inteligência artificial**, cada um especialista em uma área diferente. Juntos, eles transformam suas ideias em código funcional, testado e pronto para ser publicado.

<br/>

### Na prática, funciona assim:

> **Você escreve:** "Preciso de uma landing page para a campanha de Black Friday com formulário de captura de leads, integração com o HubSpot e otimizada para mobile"
>
> **O Hefaisto, nos bastidores:**
> 1. O agente **Sage** (produto) analisa seu pedido e cria um plano de execução
> 2. O agente **Dex** (desenvolvedor) programa a landing page
> 3. O agente **Quinn** (qualidade) verifica se a página carrega rápido, se converte bem, e se não tem bugs
> 4. O resultado é publicado como **Pull Request** — pronto para você revisar e aprovar
>
> **Tudo isso acontece automaticamente, sem você precisar fazer mais nada.**

<br/>

### O que é um Pull Request?

Se você nunca trabalhou com código, um **Pull Request** (ou PR) é como um "pedido de aprovação". O Hefaisto faz todo o trabalho e depois apresenta o resultado para você revisar antes de ir ao ar. É como um designer que mostra o layout para aprovação antes de imprimir — mas para código.

<br/>

### Para quem é o Hefaisto?

O Hefaisto foi criado especificamente para **agências de marketing digital** e seus desafios reais:

| Desafio da Agência | Como o Hefaisto Resolve |
|:---|:---|
| Cliente quer a landing page para ontem | Modo Quick executa em minutos, não dias |
| "Pode tentar em azul?" — 15 rodadas de revisão | Decisões baseadas em dados de conversão, não opinião |
| Cada cliente tem CRM diferente (HubSpot, Salesforce, RD Station...) | Integrações classificadas e mapeadas automaticamente |
| Página lenta mata o ROI dos anúncios | Performance otimizada automaticamente (Core Web Vitals) |
| Precisa de 10 variantes da mesma landing page | Templates componentizados para criação em escala |
| "Qual canal gerou essa venda?" | Atribuição multi-touch (não só "último clique") |
| Margens apertadas (10-20%) | Entregas padronizadas que reduzem custo operacional |

## Conceitos Fundamentais — Glossário para Iniciantes

Se você não tem experiência com desenvolvimento de software, esta seção explica os termos que aparecem ao longo do manual.

<br/>

### O que é um Framework?

Um **framework** é um conjunto de ferramentas e regras que facilita a criação de software. Pense nele como uma **caixa de ferramentas organizada** — em vez de procurar cada ferramenta separadamente, tudo já está no lugar certo e pronto para usar.

O Hefaisto é um framework que organiza agentes AI para que eles trabalhem juntos de forma coordenada.

<br/>

### O que é um Agente AI?

Um **agente AI** é um programa de inteligência artificial que tem uma **função específica** e pode **agir de forma autônoma**. É diferente de um chatbot comum — um agente não só responde perguntas, mas **executa tarefas concretas**.

No Hefaisto, cada agente é como um membro da equipe:
- Um planeja
- Outro programa
- Outro testa
- E assim por diante

<br/>

### O que é uma Landing Page?

Uma **landing page** (ou "página de destino") é uma página web criada com **um único objetivo** — geralmente capturar o email de um visitante, vender um produto, ou convencer alguém a se inscrever. Diferente de um site completo com várias páginas e menus, uma landing page é **focada em uma única ação**.

Exemplo: quando você clica em um anúncio do Instagram e cai em uma página pedindo seu email em troca de um e-book gratuito — isso é uma landing page.

<br/>

### O que é CRO?

**CRO** (Conversion Rate Optimization, ou Otimização de Taxa de Conversão) é a ciência de **fazer mais visitantes executarem a ação desejada** na sua página. Se 100 pessoas visitam sua landing page e 3 preenchem o formulário, sua taxa de conversão é 3%. O CRO busca aumentar esse número.

O Hefaisto aplica CRO automaticamente usando metodologias reconhecidas mundialmente.

<br/>

### O que é Git e GitHub?

**Git** é um sistema de controle de versão — ele salva o histórico de todas as alterações feitas no código, como um "Ctrl+Z" infinito e organizado. Se algo der errado, é possível voltar a qualquer versão anterior.

**GitHub** é a plataforma onde o código fica armazenado na nuvem. Funciona como um "Google Drive para código", onde equipes podem colaborar, revisar e aprovar mudanças.

<br/>

### O que é um Commit?

Um **commit** é como um "ponto de salvamento" no código. Cada commit tem uma mensagem que descreve o que foi feito. Por exemplo: `feat: adiciona formulário de contato na landing page`.

O Hefaisto faz commits automaticamente com mensagens claras e padronizadas.

<br/>

### O que é Claude Code?

**Claude Code** é uma ferramenta de inteligência artificial da Anthropic que funciona no terminal (aquela tela preta onde se digitam comandos). O Hefaisto funciona **dentro do Claude Code** — é através dele que você conversa com o Hefaisto e dá suas instruções.

<br/>

### O que é uma Story?

No mundo do desenvolvimento de software, uma **story** (ou "história") é um documento curto que descreve **o que precisa ser feito** do ponto de vista do usuário. Exemplo:

> "Como visitante da página, quero preencher um formulário com meu nome e email para receber o e-book gratuito."

O Hefaisto cria stories automaticamente para organizar o trabalho. Você não precisa escrevê-las — o Hefaisto faz isso por você.

<br/>

### O que é um Terminal?

O **terminal** (também chamado de "prompt de comando" ou "linha de comando") é uma interface onde você digita comandos de texto para o computador executar. É diferente de clicar em botões — aqui você escreve instruções diretamente.

No Windows, você pode abrir o terminal procurando por "Prompt de Comando" ou "PowerShell" no menu Iniciar. No Mac, procure por "Terminal".

## Instalação — Passo a Passo Completo

### O que você vai precisar

Antes de instalar o Hefaisto, certifique-se de ter estas ferramentas no seu computador:

| Ferramenta | Versão Mínima | O que é | Como instalar |
|:---|:---|:---|:---|
| **Node.js** | 18 ou superior | O motor que roda o Hefaisto | Baixe em [nodejs.org](https://nodejs.org) — escolha "LTS" |
| **npm** | 9 ou superior | Instalador de pacotes (vem junto com o Node.js) | Já vem com o Node.js |
| **Claude Code** | Mais recente | A IA onde o Hefaisto funciona | Siga as instruções em [claude.ai/code](https://claude.ai/code) |
| **Git** | Qualquer versão | Controle de versão do código | Baixe em [git-scm.com](https://git-scm.com) |

> **Como verificar se já tem instalado?** Abra o terminal e digite:
> ```
> node --version
> git --version
> ```
> Se aparecerem números de versão (ex: `v20.11.0`), já está instalado.

<br/>

## Primeiro Uso — Sua Primeira Conversa com o Hefaisto

### Abrindo o Hefaisto

Após a instalação, abra o **Claude Code** na pasta do projeto. Você vai ver o terminal do Claude Code esperando suas instruções.

Digite:

```
/init
```

Este comando inicia o **assistente de novo projeto** do Hefaisto. Ele vai te guiar por 5 fases:

<br/>

### Fase 1 — Descoberta do Projeto

O Hefaisto procura um documento de requisitos (chamado PRD — Product Requirements Document) na pasta do projeto. Se não encontrar, ele pede para você descrever o que precisa:

```
Hefaisto (Nova): Olá! Sou a Nova, orquestradora do Hefaisto.

  Não encontrei um documento de requisitos (PRD) no projeto.
  Sem problemas — me conte o que você precisa e eu cuido do resto.

  Descreva seu projeto:

Você: Preciso de uma landing page para promover nosso curso online
      de marketing digital. Quero uma seção hero com título impactante,
      depoimentos de 3 alunos, lista de benefícios do curso,
      e um formulário de inscrição integrado com o RD Station.
      A página precisa ser rápida e bonita no celular.
```

> **O que é um PRD?** É um documento que lista tudo o que o projeto precisa ter. Se você já tem um arquivo `PRD.md` na pasta, o Hefaisto vai usá-lo como base. Se não tem, basta descrever o que precisa — o Hefaisto entende.

<br/>

## Como Funciona — O Ciclo Autônomo

O coração do Hefaisto é o **Ciclo Autônomo**: um fluxo de 4 etapas que transforma sua ideia em código pronto para produção.

```
  ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
  │  PLAN   │ -> │  BUILD  │ -> │  CHECK  │ -> │  SHIP   │
  │         │    │         │    │         │    │         │
  │ Planeja │    │ Constrói│    │ Valida  │    │ Entrega │
  └─────────┘    └─────────┘    └─────────┘    └─────────┘
```

<br/>

### Etapa 1 — PLAN (Planejar)

**O que faz:** Analisa seu pedido, entende o que precisa ser feito, e cria um documento de planejamento (a "story") com tarefas específicas e critérios de aceitação.

**Analogia:** É como um gerente de projeto que recebe um briefing do cliente e cria um plano de ação detalhado antes de começar o trabalho.

**Exemplo de story criada:**
```
Título: Hero Section da Landing Page
Descrição: Criar a seção principal da landing page com título,
           subtítulo e botão de inscrição.

Critérios de Aceitação:
  ✓ Título exibe a proposta de valor principal
  ✓ Subtítulo contém gatilho de urgência
  ✓ Botão de CTA é visível sem rolar a página
  ✓ Layout responsivo funciona em mobile e desktop
  ✓ Carregamento em menos de 2.5 segundos
```

<br/>

### Etapa 2 — BUILD (Construir)

**O que faz:** Escreve o código, cria os componentes visuais, implementa as funcionalidades e escreve testes automatizados.

**Analogia:** É como o desenvolvedor da equipe que pega o plano e transforma em código real — a página que as pessoas vão ver e usar.

**O que acontece nos bastidores:**
- Cria os arquivos de código necessários
- Implementa o design responsivo (funciona no celular e no computador)
- Conecta integrações (CRM, analytics, etc.)
- Escreve testes para garantir que tudo funciona
- Faz commits (pontos de salvamento) com mensagens claras

<br/>

### Etapa 3 — CHECK (Verificar)

**O que faz:** Roda uma bateria de verificações automáticas para garantir que o código funciona, está limpo e a página converte bem.

**Analogia:** É como um controle de qualidade na fábrica — antes de embalar o produto, verifica se tudo está funcionando perfeitamente.

**Verificações executadas automaticamente:**

| Verificação | O que checa | Por que importa |
|:---|:---|:---|
| **Lint** | Estilo e padrões do código | Código organizado é mais fácil de manter |
| **Testes** | Se tudo funciona como esperado | Evita surpresas desagradáveis |
| **TypeCheck** | Consistência dos tipos de dados | Previne erros silenciosos |
| **CRO Score** | Se a página está otimizada para conversão | Mais conversão = mais resultados |

**Se alguma verificação falhar**, o Hefaisto tenta corrigir automaticamente (até 3 vezes). Se não conseguir depois de 3 tentativas, para e pede sua ajuda.

<br/>

### Etapa 4 — SHIP (Entregar)

**O que faz:** Publica o código no repositório (GitHub) e cria um Pull Request para sua revisão.

**Analogia:** É como embalar o produto final e colocar na mesa do cliente para aprovação — nada vai ao ar sem sua confirmação.

**O que acontece:**
1. O código é enviado para o GitHub (push)
2. Um Pull Request é criado com um resumo de tudo que foi feito
3. Você (ou sua equipe) revisa e aprova quando estiver satisfeito

> **Importante:** O Hefaisto **nunca** publica nada diretamente no ar. Ele sempre cria um Pull Request para que você revise primeiro.

## Landing Pages & CRO — Páginas que Convertem

Um dos maiores diferenciais do Hefaisto é o **scoring automático de CRO** para landing pages. Toda vez que uma landing page é criada ou modificada, o Hefaisto avalia automaticamente usando 3 metodologias reconhecidas mundialmente.

<br/>

### O que é CRO e por que importa?

**CRO** (Otimização de Taxa de Conversão) é a prática de melhorar uma página para que **mais visitantes façam o que você quer** — preencher um formulário, comprar um produto, se inscrever.

**Exemplo prático:**
- Sua landing page recebe **1.000 visitantes por dia**
- Com taxa de conversão de **2%**, você consegue **20 leads por dia**
- Se o CRO melhorar para **5%**, você consegue **50 leads por dia**
- Isso é **2,5x mais resultados** sem gastar mais em anúncios

<br/>

### Metodologia 1 — 7 Princípios CCD (Oli Gardner, Unbounce)

CCD significa **Conversion-Centered Design** (Design Centrado em Conversão). São 7 princípios criados por Oli Gardner, fundador da Unbounce, que garantem que a página seja construída para converter:

| Princípio | O que significa | Exemplo prático |
|:---|:---|:---|
| **Attention Ratio** | Uma página, um objetivo, um botão de ação | Se sua página tem 10 links diferentes, o visitante não sabe onde clicar. O ideal é ter apenas 1 caminho claro |
| **Coupling** | O anúncio e a página devem dizer a mesma coisa | Se o anúncio diz "Curso com 50% de desconto", a página deve começar com essa mesma mensagem |
| **Congruence** | Todos os elementos reforçam o objetivo | Cada imagem, texto e botão deve contribuir para a conversão — sem distrações |
| **Clarity** | Mensagem clara, sem ambiguidade | "Inscreva-se agora" é melhor que "Clique aqui para saber mais sobre nossas opções" |
| **Credibility** | Elementos que geram confiança | Depoimentos reais, logos de clientes, certificações, número de alunos |
| **Closing** | Remover todas as barreiras | Garantia de 7 dias, FAQ respondendo objeções, parcelamento disponível |
| **Continuance** | Próximo passo claro após a ação | Página de "obrigado" com próximos passos, email de confirmação |

<br/>

### Metodologia 2 — Fórmula MECLABS (Dr. Flint McGlaughlin)

É a fórmula mais respeitada para calcular a probabilidade de conversão de uma página:

```
C = 4m + 3v + 2(i-f) - 2a
```

**Traduzindo para português simples:**

| Letra | Significado | Peso | Explicação |
|:---|:---|:---|:---|
| **C** | Conversão | — | O resultado final: a probabilidade de alguém converter |
| **m** | Motivação | 4x | Quão motivado o visitante já está ao chegar na página. Fator mais importante! |
| **v** | Valor | 3x | Quão atraente é a sua oferta. O que o visitante ganha? |
| **i** | Incentivo | 2x | Existe algum motivo para agir AGORA? (desconto limitado, últimas vagas) |
| **f** | Fricção | -2x | Quantos obstáculos existem no caminho? (formulário longo, processo complicado) |
| **a** | Ansiedade | -2x | Quais medos o visitante tem? (será que é golpe? Vão me ligar?) |

> **Resumo prático:** Para converter mais, aumente a motivação e o valor percebido, crie urgência, e reduza a complicação e os medos.

<br/>

### Metodologia 3 — Core Web Vitals (Google)

São as métricas oficiais do Google para medir se uma página oferece boa experiência:

| Métrica | Limite Ideal | O que mede | Por que importa |
|:---|:---|:---|:---|
| **LCP** | < 2.5 segundos | Tempo para o conteúdo principal aparecer | Visitante desiste se demora mais que 3s |
| **CLS** | < 0.1 | Estabilidade visual (nada "pula" na tela) | Elementos que se movem causam cliques errados |
| **INP** | < 200ms | Velocidade de resposta ao toque/clique | Botão que demora para reagir parece quebrado |

> **Impacto direto no bolso:** O Google usa essas métricas no ranqueamento de anúncios. Páginas lentas pagam mais caro por clique no Google Ads.

<br/>

### Como o scoring funciona na prática

Você **não precisa fazer nada**. O processo é totalmente automático:

1. Você pede uma landing page
2. O Dex (@dev) programa a página
3. A Quinn (@qa) aplica automaticamente os 3 frameworks durante o CHECK
4. O resultado é um **score de 0 a 100** com recomendações específicas

```
Exemplo de resultado:

  CRO Score: 89/100

  ✓ Attention Ratio: 1:1 (perfeito)
  ✓ Message Match: headline alinhada com o anúncio
  ✓ Clarity: mensagem clara e direta
  ⚠ Credibility: considere adicionar logos de clientes (score: 7/10)
  ✓ Core Web Vitals: LCP 1.8s, CLS 0.02, INP 45ms
  ✓ MECLABS: boa proposta de valor, incentivo presente

  Recomendações:
  1. Adicione 3-5 logos de empresas clientes na seção de prova social
  2. Considere adicionar um selo de garantia próximo ao botão de CTA
```

## Estrutura de Arquivos — O que o Hefaisto Cria no Seu Projeto

Após a instalação, o Hefaisto adiciona duas pastas principais ao seu projeto. **Seus arquivos existentes não são modificados**.

```
seu-projeto/
│
├── .claude/                        # ⚙️ Configuração (como o cérebro do Hefaisto)
│   ├── CLAUDE.md                   # Manual interno do framework
│   ├── settings.json               # Idioma e preferências
│   ├── settings.local.json         # Permissões locais (não vai pro Git)
│   ├── rules/                      # Regras de comportamento
│   │   ├── agents.md               #   Como os agentes operam
│   │   ├── workflow.md             #   Como o ciclo funciona
│   │   ├── quality.md              #   Padrões de qualidade exigidos
│   │   └── tools.md                #   Quais ferramentas usar
│   ├── commands/Hefaisto/              # Definições de comandos
│   │   ├── init.md                 #   O comando /init
│   │   └── agents/                 #   Personalidade de cada agente
│   │       ├── hefaisto.md             #     Nova (orquestradora)
│   │       ├── dev.md              #     Dex (construtor)
│   │       ├── architect.md        #     Aria (arquiteta)
│   │       ├── qa.md               #     Quinn (qualidade)
│   │       └── product.md          #     Sage (produto)
│   └── hooks/                      # Automações internas
│
├── .hefaisto-core/                     # 🔧 Motor do framework (NÃO EDITE)
│   ├── core-config.yaml            # Configuração principal
│   ├── core/                       # Módulos do runtime
│   ├── data/                       # Base de conhecimento
│   │   ├── tool-registry.yaml      #   Catálogo de ferramentas
│   │   └── tech-presets/           #   Presets de tecnologia
│   ├── development/                # Assets de desenvolvimento
│   │   ├── agents/                 #   Definições completas dos agentes
│   │   ├── tasks/                  #   115+ workflows de tarefas
│   │   ├── templates/              #   Templates de documentos e código
│   │   ├── workflows/              #   Fluxos de trabalho
│   │   └── checklists/             #   Checklists de qualidade
│   └── cli/                        # Interface de linha de comando
│
├── docs/                           # 📄 Documentação gerada
│   └── stories/                    # Stories criadas automaticamente
│
└── (seus arquivos de código)       # 💻 Seu projeto (não é tocado)
```

<br/>

### Explicação simplificada

| O que | Para que serve | Você mexe? |
|:---|:---|:---|
| **`.claude/`** | É o "cérebro" do Hefaisto — configurações, regras e personalidade dos agentes | Raramente (só se quiser personalizar) |
| **`.hefaisto-core/`** | É o "motor" do Hefaisto — o código que faz tudo funcionar | **Nunca** (modificar pode quebrar o framework) |
| **`docs/stories/`** | Onde o Hefaisto salva os planos de cada feature | Pode ler para acompanhar o progresso |
| **Seus arquivos** | O código do seu projeto (HTML, CSS, JS, etc.) | Sim — é o seu projeto! |

> **Regra de ouro:** Você conversa com o Hefaisto pelo Claude Code. Não precisa abrir ou editar os arquivos de configuração manualmente. O Hefaisto cuida de tudo.

## Perguntas Frequentes

<br/>

<details>
<summary><strong>Preciso saber programar para usar o Hefaisto?</strong></summary>

<br/>

**Não.** O Hefaisto foi projetado para que você descreva o que precisa em linguagem natural — como se estivesse explicando para um colega de trabalho. Os 5 agentes AI cuidam de toda a programação, testes e configuração técnica.

Porém, ter conhecimento básico de tecnologia ajuda a:
- Entender melhor as entregas
- Tomar decisões mais informadas
- Revisar Pull Requests com mais segurança

</details>

<br/>

<details>
<summary><strong>O Hefaisto funciona com qualquer projeto?</strong></summary>

<br/>

O Hefaisto é **otimizado para projetos de marketing digital**: landing pages, sites, aplicações SaaS, e-commerces e integrações com ferramentas de marketing.

Ele funciona melhor com projetos web baseados em JavaScript/TypeScript, mas também suporta Go, PHP, Rust, Java e C#.

**Não é ideal para:** aplicações mobile nativas, jogos, sistemas embarcados, ou projetos que não envolvem web.

</details>

<br/>

<details>
<summary><strong>Posso usar o Hefaisto em um projeto que já existe?</strong></summary>

<br/>

**Sim!** O instalador detecta automaticamente se o projeto já tem código (chamado de modo "brownfield") e se adapta. Ele **não apaga nada** do seu projeto — apenas adiciona a estrutura do Hefaisto (pastas `.claude/` e `.hefaisto-core/`) ao lado do que já existe.

</details>

<br/>

<details>
<summary><strong>O Hefaisto faz deploy (publica no ar) automaticamente?</strong></summary>

<br/>

**Não.** O Hefaisto faz push para o repositório Git e cria Pull Requests. Você (ou sua equipe) revisa e aprova antes de publicar. O Hefaisto **nunca** faz deploy automático — essa decisão é sempre sua.

</details>

<br/>

<details>
<summary><strong>Como o Hefaisto garante a qualidade do código?</strong></summary>

<br/>

Na etapa CHECK de todo ciclo, o Hefaisto executa automaticamente:
- **Lint** — Verifica se o código segue os padrões de estilo
- **Testes** — Roda testes unitários e de integração
- **TypeCheck** — Valida a consistência dos tipos de dados
- **CRO Scoring** — Para landing pages, aplica as metodologias CCD, MECLABS e Core Web Vitals

Se alguma verificação falhar, o Hefaisto tenta corrigir automaticamente (até 3 vezes). Se não conseguir, para e pede ajuda.

</details>

<br/>

<details>
<summary><strong>Quanto custa usar o Hefaisto?</strong></summary>

<br/>

O Hefaisto em si é **open-source e gratuito**. Você precisa de:
- Uma conta no **Claude Code** (que tem seu próprio modelo de precificação)
- Opcionalmente, chaves de API para integrações externas (CRM, analytics, etc.)

</details>

<br/>

<details>
<summary><strong>O Hefaisto funciona em português?</strong></summary>

<br/>

**Sim!** O Hefaisto suporta 3 idiomas: **Português**, Inglês e Espanhol. O idioma é escolhido durante a instalação e todos os agentes se comunicam no idioma selecionado.

</details>

<br/>

<details>
<summary><strong>Posso usar apenas um agente sem o ciclo completo?</strong></summary>

<br/>

**Sim.** Você pode ativar qualquer agente diretamente com `@nome` (ex: `@dev`, `@qa`) e usar seus comandos individuais. O ciclo autônomo é o modo padrão, mas você tem controle total quando quiser.

</details>

<br/>

<details>
<summary><strong>O que é uma "story" e preciso escrevê-las?</strong></summary>

<br/>

Uma story é um documento curto que descreve **o que** precisa ser feito, **por que**, e **como saber se está pronto**. **Você não precisa escrevê-las** — o Hefaisto cria stories automaticamente quando necessário (nos modos Standard e Deep).

Elas ficam salvas em `docs/stories/` e servem como registro do que foi planejado e entregue. Você pode consultá-las a qualquer momento para acompanhar o progresso.

</details>

<br/>

<details>
<summary><strong>Como atualizo o Hefaisto?</strong></summary>

<br/>

Execute o instalador novamente no projeto:

```bash
npx @ericoautonex/hefaisto
```

O instalador detecta que o Hefaisto já está instalado e pergunta se você quer atualizar. Seus arquivos de configuração personalizados são preservados.

</details>

<br/>

<details>
<summary><strong>E se o Hefaisto fizer algo errado?</strong></summary>

<br/>

Tudo que o Hefaisto faz é registrado em commits do Git — como pontos de salvamento. Se algo der errado, é possível voltar para qualquer versão anterior. Além disso, o Hefaisto sempre cria Pull Requests para sua revisão — nada vai ao ar sem sua aprovação.

</details>

## Convenções de Commit

O Hefaisto segue o padrão **Conventional Commits** — um sistema padronizado de mensagens que facilita entender o que cada alteração fez:

| Prefixo | Significado | Exemplo |
|:---|:---|:---|
| `feat:` | Nova funcionalidade adicionada | `feat: hero section com vídeo background` |
| `fix:` | Correção de bug | `fix: formulário não enviava no mobile` |
| `docs:` | Mudança na documentação | `docs: adiciona guia de integração CRM` |
| `chore:` | Manutenção técnica | `chore: atualiza dependências` |

Quando aplicável, o commit referencia a story: `feat: seção de depoimentos [Story 1.2]`
